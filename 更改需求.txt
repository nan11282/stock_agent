# A股投资助理 — 完整实现任务书

## 一、项目概述

本地运行的个人A股投资助理。两个独立入口：
- `main.py`：CLI 对话，用户和 Agent 实时交互
- `scheduler.py`：定时任务，每天15:30收盘后自动扫描并发邮件

技术栈：Python 3.12 / Claude API / SQLite（含 FTS5 全文检索）/ ChromaDB / AKShare

## 三、数据库设计（memory.py）

### 3.1 SQLite 表结构

所有表在 `DecisionLog._init_schema()` 里统一建立：

```sql
-- 投资决策记录（append-only，不允许 UPDATE）
CREATE TABLE IF NOT EXISTS decisions (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    created_at  TEXT NOT NULL,
    stock_code  TEXT NOT NULL,
    stock_name  TEXT,
    action      TEXT,        -- buy_signal / sell_signal / hold / watch / analysis
    view        TEXT,        -- bullish / bearish / neutral
    reasoning   TEXT NOT NULL,
    price       REAL,
    ttm_yield   REAL,
    pe_pct      REAL,        -- PE历史百分位 0-100
    pe_abs      REAL,
    tags        TEXT         -- JSON 数组字符串
);

-- 自选股关注列表
CREATE TABLE IF NOT EXISTS watchlist (
    stock_code   TEXT PRIMARY KEY,
    stock_name   TEXT NOT NULL,
    reason       TEXT,
    alert_yield  REAL,       -- 股息率触发阈值（%），达到即提醒
    alert_pe_pct REAL,       -- PE百分位触发阈值，低于即提醒
    added_at     TEXT NOT NULL
);

-- 持仓表
CREATE TABLE IF NOT EXISTS positions (
    stock_code   TEXT PRIMARY KEY,
    stock_name   TEXT NOT NULL,
    cost_price   REAL NOT NULL,
    shares       INTEGER,
    position_pct REAL,       -- 仓位百分比 0-100
    tier         TEXT,       -- core / growth
    updated_at   TEXT NOT NULL
);

-- 复盘表（挂在 decisions 下，不修改原始记录）
CREATE TABLE IF NOT EXISTS retrospectives (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    decision_id     INTEGER NOT NULL REFERENCES decisions(id),
    reviewed_at     TEXT NOT NULL,
    price_now       REAL,
    outcome         TEXT,    -- correct / wrong / partial
    what_i_missed   TEXT,
    updated_view    TEXT
);

-- 对话摘要存储（供向量检索使用的原始文本）
CREATE TABLE IF NOT EXISTS episodic_docs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    doc_id      TEXT UNIQUE NOT NULL,   -- ChromaDB 对应的 id
    text        TEXT NOT NULL,
    metadata    TEXT,                   -- JSON
    created_at  TEXT NOT NULL
);

-- FTS5 全文检索索引（对 episodic_docs 的倒排索引）
CREATE VIRTUAL TABLE IF NOT EXISTS episodic_fts
    USING fts5(text, doc_id UNINDEXED, content='episodic_docs', content_rowid='id');

-- 每日扫描结果
CREATE TABLE IF NOT EXISTS scan_results (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    scanned_at  TEXT NOT NULL,
    scope       TEXT,        -- positions / watchlist / discovery
    stock_code  TEXT,
    stock_name  TEXT,
    signal      TEXT,        -- alert / opportunity / normal
    summary     TEXT
);
```

### 3.2 DecisionLog 方法

```python
    def upsert_watchlist(data: dict) -> None
    def delete_watchlist(stock_code: str) -> bool
```

### 3.3 EpisodicMemory：向量 + FTS5 混合检索

**存储结构**：
- ChromaDB 存 text + embedding 向量（高维浮点数组），用于语义检索
- SQLite FTS5 对同一批 text 建倒排索引，用于精确词检索
- 两者通过 `doc_id` 关联，写入时同步，检索时独立查询后 RRF 融合

**写入**：
```python
def save_insight(text: str, metadata: dict = None) -> str:
    # 1. 写入 ChromaDB（自动生成 embedding 向量）
    # 2. 写入 SQLite episodic_docs 表
    # 3. 更新 FTS5 索引：INSERT INTO episodic_fts(rowid, text, doc_id) VALUES (?, ?, ?)
```

**检索**（三路融合，全部使用原始 query）：

```python
def retrieve(query: str, n_results=8, top_k=4) -> list[dict]:
    # 路1：ChromaDB 向量检索 → top-n（HNSW 近似最近邻，语义感知）
    # 路2：SQLite FTS5 全文检索 → top-n（精确词匹配，对股票代码/数字敏感）
    #   SELECT doc_id, rank FROM episodic_fts WHERE text MATCH ? ORDER BY rank LIMIT ?
    # 路3：RRF 融合（K=60）
    #   score(doc) = Σ 1/(60 + rank)，两路都命中得分翻倍
    # 返回 top_k 结果，含 text / metadata / rrf_score
```

**注意**：不使用 HyDE（假设答案检索），直接用原始 query 三路融合即可。

### 4.2 写工具（严格受控）

1. 等用户说出明确操作词（保存 / 删除 / 加入 / 移除 / 更新）
2. 以结构化格式展示将要执行的内容
3. 等用户回复"确认"后才调用

**工具列表**：
```

upsert_watchlist(stock_code, stock_name, reason, alert_yield, alert_pe_pct)
```

每个写工具的 description 必须包含：
```
【严格限制】只有用户明确说"XXX"时才能调用。
调用前必须展示将要执行的内容，等用户回复"确认"后执行。
```

## 十一、注意事项

1. FTS5 中文检索需要先分词再 MATCH，在写入时对 text 分词后存一列，或在查询时对 query 分词后用 `OR` 连接：
   ```sql
   WHERE episodic_fts MATCH '招行 OR 招商银行 OR 股息率'
   ```
   使用 jieba 分词，token 之间用空格连接传给 MATCH。

2. decisions 表严格禁止 UPDATE，任何修改走 retrospectives 追加。

3. 写工具的 description 里的【严格限制】文字不得简化，这是控制 Agent 行为的关键约束。

4. scheduler.py 和 main.py 是两个独立进程，共享同一个 SQLite 文件，SQLite 的 WAL 模式可以支持并发读，在 `_init_schema` 里加上：
   ```python
   self.conn.execute("PRAGMA journal_mode=WAL")
   ```
